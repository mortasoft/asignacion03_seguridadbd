01 — Protección

¿Cómo Estamos Protegidos?

En lugar de concatenar el input del usuario al SQL, usamos marcadores de posición ?. El driver ODBC se encarga de escapar cualquier carácter peligroso automáticamente.

☠️

v1 Vulnerable

Usa f-strings para concatenar el input al SQL. El atacante puede romper la sintaxis e inyectar código.

🔒

v2 Segura (esta)

Usa ? como marcador. Los datos van SEPARADOS del código SQL. Imposible romper la sintaxis.

Python — Query Parametrizada 
# El ? es un marcador de posición, nunca se convierte en SQL
query = "SELECT name, database_id FROM sys.databases WHERE name LIKE ?"

# Los datos se pasan como argumento separado — el driver los escapa
cursor.execute(query, (f"{apellido}%",))

# Si apellido = "' OR '1'='1'--"
# La query busca literalmente la cadena: ' OR '1'='1'--
# Resultado: [] — no existe esa base de datos
02 — Mecánica

Por Qué Esto Es Seguro

El driver ODBC separa completamente el código SQL de los datos del usuario.

1

El SQL se envía primero (sin datos)

El servidor recibe la estructura de la query con el marcador ? y la compila. Ya sabe qué es código y qué será dato.

SELECT name, database_id FROM sys.databases
WHERE name LIKE ?  ← el servidor compila esto primero
2

Los datos se envían después (como valores)

El driver envía el input del usuario como un parámetro de datos, no como código SQL. El servidor lo trata como texto plano.

# Aunque el atacante envie esto:
input = "' OR '1'='1'--"

# El servidor lo trata como valor de texto, no como SQL:
WHERE name LIKE "' OR '1'='1'--"  ← string literal, no SQL
3

Resultado: el ataque no tiene efecto

El servidor busca una base de datos cuyo nombre sea literalmente ' OR '1'='1'--. Como no existe, retorna una lista vacía [].

03 — Demo

Prueba la Protección

Los mismos payloads de ataque que funcionan en v1 son completamente inofensivos aquí.

Consulta Normal
Funciona igual que v1 para entradas legítimas.
GET http://localhost:5002/cliente?apellido=m
↗ Probar Resultado: master, model, msdb
Ataque OR 1=1 — Neutralizado
El payload de inyección es tratado como texto literal. Devuelve [] porque no existe esa base de datos.
GET http://localhost:5002/cliente?apellido=' OR '1'='1'--
↗ Probar Seguro: devuelve []
Ataque UNION SELECT — Neutralizado
La inyección UNION tampoco funciona. El UNION se interpreta como parte del nombre a buscar.
GET http://localhost:5002/cliente?apellido=x' UNION SELECT name, database_id FROM sys.databases--
↗ Probar Seguro: devuelve []
¿Quieres ver los mismos ataques funcionar?

En la versión vulnerable (v1), los mismos payloads devuelven todas las bases de datos del servidor.
